---
name: prd
description: 基于业务描述，生成结构化 PRD 文档，包含用户故事、验收标准、业务规则和开放问题。
---

## 使用方式

```md
/prd 功能描述
```

示例：

```md
/prd 产品管理：运营人员需要能够维护产品信息，包括上下架、价格调整和库存管理
/prd 工单流转：维修人员接收工单、填写处理记录、关闭工单，管理员可查看全部
```

激活 `@product-manager` 代理

请严格按照以下“产品经理工作流”开始执行任务：

### 📋 第一阶段：需求挖掘与访谈

1. 询问用户本次想要开发的产品或功能是什么。
2. 针对用户的初步想法，从“目标用户”、“核心痛点”、“使用场景”、“商业价值”四个维度进行追问，确保需求清晰。
3. 在需求不明确的情况下，请用户进行详细追问，直到需求明确为止。
4. 确定需求范围，根据需求范围，确定需求优先级。

### 📝 第二阶段：结构化梳理

1. 将沟通确认后的需求，转化为结构化的功能列表（Feature List）。
2. 拆解核心用户故事（User Stories），并为每个故事编写验收标准（Acceptance Criteria）。

### 🚀 第三阶段：产品设计

1. 根据功能列表，进行产品设计，确定产品架构。
2. 确定产品交互设计，并生成交互设计稿。
3. 确定产品流程设计，并生成流程设计稿。
4. 确定产品数据模型，并生成数据模型。
5. 确定产品用户角色，并生成用户角色。
6. 确定产品权限控制，并生成权限控制。6. 确定产品权限控制，并生成权限控制。

### 第四阶段：文档编写

在等待用户确认所有需求与需求范围之后，生成完成的结构化 PRD 文档， 包含：

- 背景与目标
- 功能清单 （MoSCow 优先级）
- 用户故事 （每个Story 含 3+ 条 AC）
- 业务规则 （含核心、辅助、数据、权限矩阵）
- 异常处理
- 非功能需求
- 开放问题
- 变更日志

### 第五阶段：文档审核

在输出 PRD 前，进行自检，确保文档质量符合以下检查项：

1. 背景与目标：确保背景与目标清晰，无歧义。
2. 功能清单：确保功能清单清晰，无歧义。
3. 用户故事：确保用户故事清晰，无歧义。
4. 业务规则：确保业务规则清晰，无歧义。
5. 异常处理：确保异常处理清晰，无歧义。
6. 非功能需求：确保非功能需求清晰，无歧义。
7. 开放问题：确保开放问题清晰，无歧义。
8. 变更日志：确保变更日志清晰，无歧义。
9. 架构设计：确保架构设计清晰，无歧义。
10. 流程设计：确保流程设计清晰，无歧义。
11. 数据模型：确保数据模型清晰，无歧义。
12. 用户角色：确保用户角色清晰，无歧义。
13. 权限控制：确保权限控制清晰，无歧义。

## 第六阶段：确认并写入文件

展示 PRD 草稿，等待用户确认后写入：

```
docs/prd/{功能英文名}-prd.md
```

同时输出用于跨团队沟通的摘要版本文件：docs/prd/{功能英文名}-prd-summary.md

内容包含以下内容：

```
## 功能摘要：{功能名称}

**核心需求**：{1-2 句话描述}
**用户群体**：{用户角色}
**关键功能**：
- Must Have：...
- Should Have：...
**验收标准数量**：{N} 条
**开放问题**：{N} 个待确认
```

## 第七阶段：生成后续任务建议

```
## 建议后续步骤

1. 运行 /ui-spec {功能名称} → 生成交互设计规范
2. 运行 /fed-arch {功能名称} → 搭建前端架构
3. 运行 /api-design {功能名称} → 设计接口契约
4. 运行 /test-plan {功能名称} → 生成测试计划
4. 运行 /sprint {功能需求} → 拆解开发迭代计划
```
